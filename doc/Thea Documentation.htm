Background

Currently Thea supports a number of reasoning methods. We need to evaluate different approaches and build an common interface/layer around them.

Thea should implement the OWL2  RL Rules, a set of first order implications that take the form

if AntecentedList then ConsequentAxiomList

Originally in RL rules the antecedents and consequents are represented as RDF triples, but in Thea the triples have been transformed already to OWL axioms.

For example the rule for the transitivity of subproperty relation (see rule scm-spo, in table 9, par 4.3 of the  RL Rules ) can be written as a (prolog) rule: 

subPropertyOf(X,Y) :-
        subPropertyOf(X,Z),
        subPropertyOf(Z,Y).

After parsing and OWL/RDF URI (file or http based resource), Thea generates OWL2 axioms from the RDF triples. These axioms are stored as axiom/1 facts.

Requirements

The reasoner should:

1.      Represent the OWL RL Rules in a suitable prolog format.

2.      Have an engine to reason over existing axioms using these rules. i.e determine if consequent of the rule holds (and under what variable bindings) given existing axioms.

3.      Generate new axioms (e.g. inferred axioms) and be able to give an explanation of why the new axioms where inferred (by which rule and which existing axioms as antecedents).

4.      Be able to separate inserted for inferred axioms and clear possible materialisation (assertions) of inferred axioms.

5.      OWL2 axioms in the prolog database are also stored as axiom/2 facts: Axiom(Ontology, Axiom). The reasoner should be able to reason over specific Ontology(-ies) only.

6.      Reasoning should be inceremental: When new axioms are added it should be able to infer axioms based on the newly added axioms only (for efficiency). This is similar to bottom–up triggering rules in production systems.

Thea Open Calais wrapper module

1 Introduction

Open Calais is a Web service provided by Thomson Reuters. Open Calais analyses content using NLP and semantic tecnniques and returns an RDF based response containing entities and relationships identified in the source and associated metadata.

thea_opencalais is a Prolog wrapper for accessing Open Calais and process the results. It is an application of Thea prolog library for OWL2 Ontologies. thea_opencalais uses the OpenCalais ontology and parses the service response first into Ontology axioms and finally into prolog terms and predicates, accessible from within a prolog program.

2 Example

  1. Get your Open Calais license key and assert it as a prolog fact: 
2.       :- assert(thea_opencalais:open_calais(license('Your open calais license key'))).
  1. Load the Open Calais Ontology into Prolog using Thea. 
4.      :- owl_parse_rdf('owl.opencalais-4.3.xml.owl',[imports(false),clear(complete)]).

The Ontology in this example has been already downloaded in the working directory. Originally it can be found here

  1. Post the content of a Wikipedia page about papal visits to the Open Calais REST service 
6.      :- oc_rest(http('http://en.wikipedia.org/wiki/List_of_journeys_of_Pope_Benedict_XVI'),'',_X).
  1. Use Prolog provided prolog predicates to examine Markup Elements (Entities and Relationships) in the result

  1. You can also write custom predicates to query the resulted database of Markup Elements e.g.

3 Predicate reference

oc_rest(+Request, +Params:string, -Result) is det

Calls Open Calais Rest service and posts the Request. Requires a valid Open_Calais license key in a dynamic open_calais(license(License)) to be asserted.

Parameters:

Request

- can be any of the following:

http(URL)

file(Filename)

text(Atom)

Params

- is a string encoded XML see description in http://www.opencalais.com/APICalls

Result

- is the RDF response of Open_Calais see http://www.opencalais.com/documentation/calais-web-service-api/interpreting-api-response/rdf

oc_entity(?I, ?C, -PV:list, +Instance:list, +Resolutions:list) is nondet

Returns the markup element entities in the response to an Open Calais request

Parameters:

I

- is the returned markup entity ID (IRI)

C

- is the entitie's markup type

PV

- list of property value pairs for the entity

Instance

- List of instance_info terms for the specific entity

Resolutions

- List of resolution terms for the specific entity

oc_relation(?I, ?C, -PV:list) is nondet

Returns the markup element relations in the response to an Open Calais request

Parameters:

I

- is the returned markup relation ID (IRI)

C

- is the relation's markup type

PV

- list of property value pairs for the relation

oc_resolution(?R, ?C, -PV:list) is nondet

Returns the resolutions in the response to an Open Calais request

Parameters:

R

- is the returned resolution ID (IRI)

C

- is the resolution's type

PV

- list of property value pairs for the resolution
